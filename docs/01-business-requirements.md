# 01. 要求仕様書（Business Requirements）

このプロジェクトが「なぜ存在するのか」「誰のために何を解決するのか」を定義する。

## 背景

GitHub には自分のリポジトリ全体を俯瞰できる「インデックスページ」が存在しない。
そのため、自分の GitHub の中にどこに何があるか把握しづらく、過去の作業の参照や他者への紹介がしにくい。

GitHub Pages（github.io）を活用し、「まずはここを見に行けば良い」という入口となる
静的インデックスサイトを HTML として公開することで、この課題を解決する。

## 目標（ゴール）

- 自分の **public リポジトリ** を一覧表示できる
- 各リポジトリの「名前」「概要（description）」「URL」を一覧で参照できる
- GitHub Pages（プロジェクトページ形式）で公開する
- リポジトリの追加・更新は自動的に反映される（手動メンテ不要）

## ステークホルダー

| 立場 | 関心事 |
|------|-------|
| 開発者（自分） | 自分のリポジトリを俯瞰できる入口がほしい |
| 訪問者（採用担当・知人等） | この人がどんなものを作っているか一覧で見たい |

## スコープ

### In Scope

- 自分の **public** リポジトリのみを対象とする
- 表示項目は「リポ名」「概要（description）」「リポ URL」の 3 つに限定
- GitHub Actions の cron による定期再ビルドで最新化
- GitHub Pages（プロジェクトページ：`username.github.io/my-github-index/`）で公開

### Out of Scope

- **private リポジトリの表示**（別プロジェクト・別リポジトリで対応予定）
- スター数・言語・最終更新日などの追加メタデータ表示
- ログイン機能・ユーザーごとの表示切り替え
- リポジトリの編集・削除などの操作機能
- コメント・いいね等のソーシャル機能

## 制約

- **ホスティング**: GitHub Pages のみ（サーバーサイド処理は不可、静的ファイルのみ）
- **GitHub API**: 無認証で利用（rate limit 60 req/hr）。ビルド時のみ呼び出すため問題なし
- **コスト**: 無料運用を前提とする
